# 场景适配

## 目标

这份文档回答一个实际问题：

高性能导向的设计，怎么覆盖复杂业务场景，而不把架构拉回到“通用状态系统 + 通用渲染系统”。

这里重点处理三类场景：

- 长列表
- 跨边界通信
- 异步更新

## 长列表

### 问题

普通 `KEYED_LIST` 足够处理一般列表，但不够处理超长列表。

如果对长列表仍然按普通 region 做 mount / unmount，就会遇到两个问题：

- DOM 创建和销毁太多
- 滚动过程中局部更新成本失控

### 设计

长列表走 `VIRTUAL_LIST`。

它不是“一个列表加一个判断”，而是特化 region。

这个 region 至少要做三件事：

1. 维护可见窗口和 overscan
2. 复用已创建的 DOM 节点
3. 把 item 数据重绑定到局部 cell slot

### 约束

- 编译器只负责标记可虚拟化边界和 slot
- 窗口大小、overscan、节点池策略由运行时决定
- 列表项依赖限定在 region 局部 slot 内，不向外扩散

### 明确不采用

- 编译期写死窗口大小
- 每次滚动都整批销毁和重建节点

## 跨边界通信

### 问题

复杂业务里，Region 和 Instance 不可能完全隔离。

如果没有通信机制，业务会很难写。
如果直接上全局 signal，共享范围又会失控。

### 设计

跨边界通信走 `Channel + 显式订阅表`。

这里不是做一个泛化 EventBus，而是做一套受限通信面：

- 发送方声明写入哪个 channel slot
- 接收方声明订阅哪个 channel slot
- scheduler 在受控时机消费消息

### 约束

- 订阅关系必须显式声明
- channel 只负责传递消息，不负责开放任意共享状态
- 禁止全局 signal 直接承担跨边界通信

### 明确不采用

- 任意地方都能读写的全局 signal
- 发送方直接改接收方实例内部字段
- 缺少订阅声明的广播模型

## 异步更新

### 问题

接口请求、定时器、后台刷新都会带来异步更新。

如果这些更新不进统一 scheduler，会出现：

- 更新时机不稳定
- 旧结果覆盖新状态
- 主线程被低价值更新挤占

### 设计

异步更新进入统一 scheduler，并按 lane 调度。

推荐 lane：

- `SYNC_INPUT`
- `VISIBLE_UPDATE`
- `DEFERRED`
- `BACKGROUND`

同时要有版本确认机制：

- 发起异步任务时记录 `requestVersion`
- 结果返回时对比 `versionSlot`
- 只有匹配当前版本的结果才能提交

### 约束

- 优先级按 UI 影响定，不按任务来源定
- 网络请求不天然等于低优先级
- 定时器也不天然等于后台任务

### 明确不采用

- “异步任务统一低优先级”
- 请求返回后直接写状态，不做版本校验
- 把异步更新当成独立于 scheduler 的旁路

## 收敛原则

这三类场景都遵守同一条原则：

不是给运行时再加一套通用能力，而是在现有 slot / region / scheduler 体系内增加特化边界。

也就是说：

- 长列表靠特化 region
- 通信靠显式 channel
- 异步靠 lane 和版本确认

这样能补场景，不会把核心架构打散。
