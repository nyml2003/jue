# 风格 DNA

这份文档不是重复 `code-style-spec.md`。

`code-style-spec.md` 讲的是边界和硬规则。
这份文档讲的是这套代码为什么“看起来顺”，以及写新代码时应该保留什么味道。

如果要用一句话概括当前仓库的风格，它更像：

`系统代码 > 框架糖衣 > 业务脚本`

也就是说：

- 先把边界和数据结构说清楚
- 再把状态转移和错误路径写清楚
- 最后才是语法层面的简洁和好看

## 1. 先写事实，再写包装

这套代码不喜欢先造抽象名词，再往里塞逻辑。

它更常见的顺序是：

1. 先把真实数据结构列出来
2. 先把真实状态分支列出来
3. 再决定需不需要包一层 helper 或 builder

所以你会经常看到：

- `bindingOpcode`
- `bindingNodeIndex`
- `regionBranchRangeCount`
- `signalToBindings`

这种名字不“优雅”，但非常诚实。

## 2. 名字必须直接暴露职责

当前风格不喜欢：

- 模糊词
- 情绪词
- 自造词
- “增强版”“高级版”“优化版”这类悬空命名

更偏好：

- 动作名函数
- 结构名字段
- 能看出输入输出边界的类型名

例如：

- `dispatchBinding`
- `lowerBlockIRToBlueprint`
- `createBlueprintBuilder`
- `BindingDispatchError`

这里的共同点是：

- 看名字就知道它做什么
- 看名字就知道它属于哪一层
- 看名字就知道失败时大概会返回什么

## 3. 控制流要像状态机，不像魔术

当前代码读起来顺，很大原因是控制流非常直。

常见模式是：

1. 先取当前 slot / index / ref
2. 立即检查非法状态
3. 非法就返回结构化错误
4. 合法才继续做 host patch 或 lowering

这让代码非常像“状态机展开”，而不是“聪明技巧集合”。

它不追求把逻辑压成最少行数。
它追求的是：

- 每个 guard 都说得清
- 每个失败点都能解释
- 每一步都能单测

## 4. 可预期失败要显式建模

这套风格明显偏爱 `Result`。

不是因为 `throw` 不能用。
而是因为这里的大部分失败都属于：

- 输入非法
- 引用缺失
- region 定义不合法
- 编译输入超出当前支持面

这些都不是“程序炸了”，而是“当前状态不满足契约”。

所以更适合写成：

- `code`
- `message`
- `ok / value / error`

这种写法的好处是：

- 测试直接锁定错误语义
- 调用方被迫处理失败分支
- 错误不会悄悄穿透多层栈

## 5. 边界比复用更重要

这套代码不是不想复用。
它只是把“边界不能脏”放在“少写几行”前面。

所以它宁愿：

- 在 `runtime-core` 写一套明确的 dispatch
- 在 `web` 写一套明确的 host 映射
- 在 `compiler` 写一套明确的 lowering

也不愿意搞一个大而全的跨层 `utils`。

这也是为什么仓库里没有那种特别显眼的：

- `helpers.ts`
- `misc.ts`
- `shared-utils.ts`

因为这类文件很容易把层之间的职责搅混。

## 6. 热路径写法要有“硬度”

这个仓库的运行时代码很强调“硬”。

这里的“硬”不是难读。
而是：

- 数据结构固定
- 字段稳定
- 分支有限
- 输入输出明确

所以热路径偏好：

- typed array
- 平行数组
- 固定字段对象
- 整数索引

不偏好：

- 深层对象图
- 到处挂 metadata
- 靠 `Map` / `Set` 串热路径
- 同一个数组里混很多种状态类型

你会感觉这些代码在“拒绝浪漫”，但它们很适合长期优化。

## 7. 注释只解释不能从代码直接看出的事

当前风格并不依赖注释来解释“这行代码在做什么”。

它更适合写三类注释：

- 为什么这样分层
- 为什么这个顺序不能变
- 为什么这里故意不支持某种写法

换句话说：

- 行为解释优先靠命名
- 设计约束才值得写注释

如果一段代码必须靠长注释才读得懂，通常意味着：

- 命名不够直接
- 职责切分不够清楚
- 控制流还可以继续收紧

## 8. 测试不是补丁，是规格

这个仓库的测试写法也很统一。

它不是“随便测一下能跑”。
它更像是在写小型契约：

- 给一个最小输入
- 要么得到一个确定输出
- 要么得到一个确定错误

所以测试名通常非常直给：

- `lowers ... into ...`
- `reports ...`
- `rejects ...`
- `compiles ... into ...`

这种测试风格的优点是：

- 读测试就能知道当前支持面
- 读测试就能知道当前故意不支持什么
- 回归时很容易判断是功能变化还是契约破坏

## 9. TSX authoring 也保持显式

即使在 examples 里，这套风格也不喜欢把一切都塞进一个 JSX 表达式里。

更常见的是：

- 先定义一批 `signal(...)`
- 再在 TSX 里一项项接进去

这会让代码有点“冷”。
但好处很明显：

- 每个动态点都能直接定位
- 编译器输入边界更清楚
- example 本身就能当 compiler 回归面

它不是在追求 React 式的自然书写感。
它是在追求“authoring 输入能稳定映射到 runtime 契约”。

## 10. 不抢跑，不偷未来

这套风格有一个很明显的自制力：

不为了显得完整，提前塞进未来能力。

例如当前明确不支持的东西，会直接说不支持，而不是偷偷兜底：

- fragments
- spread props
- hooks
- 组件调用
- template 内复杂结构

这让代码看起来很克制。

但这种克制很重要，因为它避免了两个坏结果：

- 用兼容层把后端契约污染掉
- 为了“先支持一点”留下长期还债点

## 11. 写新代码时，优先保持这 12 条感觉

1. 先问职责边界，再写实现。
2. 先问数据结构是否稳定，再写语法糖。
3. 函数名用动作，字段名用结构，不写情绪词。
4. 可预期失败优先 `Result`，不要先想到 `throw`。
5. 先 guard，后主逻辑。
6. 每个分支都要能回答“保护了什么”。
7. 不用大而全 helper 跨层复用。
8. 热路径少对象图，多索引和稳定数组。
9. 注释只解释约束，不解释显然行为。
10. 测试名直接写成能力或失败条件。
11. authoring 示例也要当契约输入写，不要当 UI 草图写。
12. 对当前不支持的东西，明确拒绝，不偷偷兼容。

## 12. 一句话版本

如果以后要继续保持这套味道，可以记住这句：

`直接命名，显式建模，短控制流，硬边界，少魔法。`
