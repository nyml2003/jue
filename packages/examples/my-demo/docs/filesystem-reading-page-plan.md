# My Demo 文件系统阅读页方案

## 1. 目标

这份文档定义 `packages/examples/my-demo` 里文件系统式阅读页的第一版落地范围。

目标不是一次把原始 SysFolio 设计完整实现出来。

目标是先落一个版本，这个版本需要同时满足：

- 符合 `jue` 当前 authoring / compiler 的约束；
- 能证明页面骨架和交互方向是成立的；
- diff 小、容易验证；
- 不提前引入当前框架还支撑不稳的抽象层。

## 2. 当前范围

这次只做：

- 左侧 `FileTree`
- 右侧 `ContentPane`
- 左侧支持嵌套文件夹
- 支持选中文件
- 右侧只显示文本内容

这次明确不做：

- `ContextPanel`
- 树内搜索
- 阅读进度恢复
- 目录摘要视图
- 文档目录
- 主题切换
- 未支持文件类型的统一空态
- 为了追求通用性而硬做的递归树 authoring

## 3. 页面形态

### 3.1 布局

先按桌面端两栏布局实现：

- 左栏：窄导航树
- 右栏：主阅读区

建议比例：

- `FileTree`：`280px` 到 `320px`
- `ContentPane`：剩余宽度

### 3.2 用户路径

第一版用户路径保持简单：

1. 用户进入页面，先看到左侧文件系统式树形导航。
2. 用户展开文件夹。
3. 用户点击一个文本文件。
4. 右侧更新为该文件的标题、路径和正文。

第一版不要再并行引入第二套导航心智。

## 4. 建议的 v0 组件

### 4.1 `FileTree`

职责：

- 展示文件夹 / 文件层级
- 支持文件夹展开和收起
- 支持文件选择
- 保持当前选中项可见

第一版刻意收窄：

- 不做内置搜索
- 不做次级元信息
- 不做拖拽
- 不做虚拟列表
- 不做右键菜单

### 4.2 `ContentPane`

职责：

- 展示当前文件路径
- 展示当前文件标题
- 展示纯文本正文段落

第一版只承载类似文章的文本内容。

### 4.3 数据模型

demo 里推荐先用最小模型：

- `folder`
- `article`

建议节点形状：

```ts
type DemoNode =
  | {
      kind: "folder";
      id: string;
      name: string;
      children: readonly DemoNode[];
    }
  | {
      kind: "article";
      id: string;
      name: string;
      title: string;
      pathLabel: string;
      body: readonly string[];
    };
```

这个形状已经足够证明交互模型，不需要过早把未来的多内容类型一起塞进来。

## 5. 面向 `jue` 的实现策略

### 5.1 稳妥路径

在当前 `jue` 里，最稳妥的路径是：

- 用显式 `signal()` 管状态
- 事件处理器全部用命名函数
- 只用 host primitives
- JSX 表达式保持简单
- 不依赖复杂组件组合能力

需要维护的状态：

- 当前选中的文件 id
- 当前展开的文件夹 id 集合
- 如有必要，再维护右侧显示用的标题 / 路径 / 正文内容

### 5.2 树渲染策略

不要先假设“通用、可点击、可递归的数据驱动树”就是当前 `jue` 下的最佳第一版实现。

更稳妥的选择有两个：

1. 先写一个固定 demo 层级，用显式嵌套分支表达树。
2. 也可以保留数据模型，但渲染层先用显式 helper 分支，不直接走 `List` 驱动的交互式树行。

原因很直接：

- `List` 是存在的；
- 但 `List` 模板回调内部挂事件处理器，目前 compiler 明确不支持。

这意味着“最自然的可点击树 authoring 写法”现在还不是稳定主路径。

### 5.3 内容区渲染策略

右侧先做普通文章视图：

- 一行路径
- 一个文档标题
- 多段正文，每段单独渲染成 `Text`

第一版不需要 markdown parser。

## 6. `jue` 现在已经足够支持的能力

根据当前仓库文档和 compiler 测试，下面这些能力可以放心作为第一版基础：

- PascalCase host primitives：`View`、`Text`、`Button`、`Input`、`Image`、`ScrollView`
- 宿主无关事件：`onPress`、`onInput`、`onFocus`、`onBlur`、`onScroll`
- 显式 `signal().get() / set() / update()`
- 命名函数事件处理器
- 单根 JSX 返回
- `Show`、`List`、`VirtualList`
- 简单的文本页面和 signal 驱动内容切换

## 7. `jue` 现在还不支持得很干净的能力

下面这些约束和这个页面直接相关。

### 7.1 `List` 里的交互项

当前 compiler 测试明确拒绝在 `List` 或 `VirtualList` 模板回调里放事件处理器。

实际影响：

- 用 `List` 渲染一棵通用可点击树，今天不是稳定路径。

### 7.2 内联事件处理器

当前 guaranteed subset 偏向命名函数 handler。

实际影响：

- 避免写 `onPress={() => ...}` 这种内联 arrow。

### 7.3 JSX spread props

spread props 现在不支持。

实际影响：

- props 和 style 都应该显式写出来。

### 7.4 组件 / 成员式 JSX 组合

当前 compiler 测试对更自由的 component / member JSX tag 支持并不稳定。

实际影响：

- 不要把这页默认当成已经具备 React 那种成熟组件组合面。

### 7.5 过宽的模板表达式自由度

当前 compiler 支持面仍然偏窄，本质上还是 compiler-safe subset。

实际影响：

- 复杂值先在 script 部分算好；
- JSX 里只消费简单结果。

### 7.6 `Portal` 和 `Boundary`

这两个名字虽然存在，但明确还没实现。

实际影响：

- 这页方案里不应该引入它们。

## 8. 合理诉求和不合理诉求

### 8.1 当前合理的诉求

对于 `my-demo`，下面这些诉求是合理的：

- 先做一个简单的两栏阅读页
- 左边支持嵌套目录
- 支持文本文件选中
- 右侧跟着选中结果更新
- 保持代码显式、直接、规模可控
- 高级抽象先推迟，等 compiler 支持面补上

### 8.2 合理但当前受框架限制的诉求

下面这些是合理产品诉求，但还不是当前 `jue` 主路径里很干净的 authoring：

- 通用数据驱动的可点击递归树
- 可复用的树节点组件，并在列表模板里自由挂事件
- 更自然的页面组件组合
- 更丰富的文档结构和更宽的 JSX 表达式支持

这些不是产品想法有问题，而是当前框架还没把这条路径收稳。

### 8.3 现阶段不合理的诉求

如果目标是留在当前 `jue` 约束里，下面这些诉求现在是不合理的：

- 一边要求完全通用的文件系统 UI，一边要求不接受任何面向框架现状的妥协
- 把 `array.map()`、spread props、inline handlers、DOM-like authoring 当作默认应有能力
- 把所有缺失的 authoring 模式都当成页面层问题，而不是 compiler / frontend 的能力缺口

## 9. 推荐实施顺序

后面开始实现时，建议按这个顺序推进：

1. 先把登录 demo 壳替换成两栏阅读布局。
2. 加一份小型硬编码内容树。
3. 给文件夹加显式展开 / 收起 handler。
4. 加文件选择和右侧文本渲染。
5. 最后再补间距、排版和选中态样式。
6. 做完这些之后，再决定树的 authoring 要不要泛化。

## 10. 结论

对 `my-demo` 来说，正确的第一步是：

- 先把阅读页做出来；
- 但实现方式必须尊重 `jue` 当前 compiler-safe subset；
- 不要过早强推更抽象的通用树架构。

所以第一版有意比原始设计稿更简单，而且这种简化是有技术依据的。
